# rspack/rslib Bug: Unsafe ESM output for CJS external modules

## Summary

When rslib/rspack externalizes a CJS dependency and emits ESM output, it generates `import * as X from 'cjs-module'` with direct property access (`X.namedExport`). At runtime in Node.js, this fails because `cjs-module-lexer` cannot detect named exports from CJS modules that use dynamic export patterns (like the `_export()` helper generated by swc/babel).

## Steps to Reproduce

```bash
pnpm install
pnpm build
pnpm test:esm   # ❌ FAILS
pnpm test:cjs   # ✅ Works
```

## Expected Behavior

The ESM output should work at runtime. Either:
1. Keep it as `import * as X` but access via `X.default.prop` (since Node.js puts CJS module.exports on `.default`)
2. Or emit `import X from '...'` + `X.prop` (default import gives the whole module.exports)

## Actual Behavior

The ESM output:
```js
import * as __rspack_external_fake_cjs_lib from "fake-cjs-lib";
const sym = __rspack_external_fake_cjs_lib.MY_SYMBOL;       // undefined!
const fn = __rspack_external_fake_cjs_lib.helperFn;         // undefined!
```

At runtime in Node.js ESM, `import * as X from 'cjs-module'` creates a namespace where:
- `X.default` = the full `module.exports` object (CJS exports live here)
- Named exports are only available if `cjs-module-lexer` can statically detect them

For CJS modules using the dynamic `_export()` helper pattern (swc/babel output), `cjs-module-lexer` cannot detect named exports, so they're only available on `.default`.

## The CJS Module Pattern (from swc/babel)

```js
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
function _export(target, all) {
    for(var name in all) Object.defineProperty(target, name, {
        enumerable: true,
        get: all[name]
    });
}
_export(exports, {
    MY_SYMBOL: function() { return MY_SYMBOL; },
    helperFn: function() { return helperFn; }
});
```

`cjs-module-lexer` can recognize `exports.X = ...` and `module.exports = { X }` but NOT this `_export()` helper pattern.

## Real-World Impact

This affects `@lynx-js/tailwind-preset` which depends on tailwindcss internal modules (`tailwindcss/lib/lib/setupContextUtils.js`, etc.) that use exactly this pattern. The built ESM output fails with:

```
SyntaxError: The requested module 'tailwindcss/lib/lib/setupContextUtils.js' does not provide an export named 'INTERNAL_FEATURES'
```

## Environment

- Node.js: v24.12.0
- rslib: 0.22.0 / 0.19.6
- OS: macOS
